# Accessibility Tree Observation Channel

Channel epic: **#2136** — `epic(jet): parity — accessibility tree observation channel`
Umbrella: **#2133** — perceptual parity harness
Sub-issues: **#2158, #2159, #2160, #2161, #2162, #2163**

## Goal

`jet` paints all visible UI to a single `<canvas>` driven by React-in-WASM and a
WebGPU backend. A canvas is opaque to assistive technology: NVDA, VoiceOver,
JAWS, and TalkBack see exactly one node — an unlabeled graphic — and read
nothing useful aloud. Without an explicit semantics layer, a blind or
low-vision user literally cannot operate the app, regardless of how
pixel-perfect the visible UI is. This is the same problem Flutter Web hit
with CanvasKit and solved with the engine's `semantics/` subsystem, which
builds a hidden DOM mirror that screen readers walk in place of the canvas.

The accessibility tree observation channel is the conformance harness that
proves jet's hidden ARIA mirror reads aloud the same way react-dom + MUI
does for the same JSX. Concretely, for every fixture in the parity corpus
we capture three artifacts on both renderers — (1) axe-core violation set
against WCAG 2.1 AA, (2) Chrome DevTools Protocol `Accessibility.getFullAXTree`
snapshot reduced to `{role, name, value, state}` tuples, (3) live-region
mutation trace — and diff jet against the react-dom oracle within a
per-fixture tolerance whitelist. A channel failure means a real screen-reader
user hears the wrong thing, hears nothing, or cannot find a control; we do
not consider "looks identical on screen" a green build until the AX tree
diff is clean.

## Architecture

jet's renderer is structurally identical to Flutter Web's CanvasKit path: a
single visible canvas plus a sibling DOM subtree whose only purpose is to
expose semantics to the platform a11y APIs. The subtree is built by an
emitter that walks the React fiber tree at commit time and converts each
mounted widget into one ARIA-bearing element (or a small fragment for
composite widgets).

```
<canvas id="jet-root" />                      <!-- visible pixels, opaque to AT -->
<div id="jet-semantics" aria-hidden="false">  <!-- shared with focus channel -->
  <button aria-label="Login" tabindex="0">Login</button>
  <ul role="listbox" aria-label="Country">
    <li role="option" aria-selected="true">Taiwan</li>
    <li role="option">United States</li>
  </ul>
  <div role="status" aria-live="polite" />    <!-- toasts / form errors -->
</div>
```

The pattern follows Flutter's `SemanticsBinding` + `SemanticsObject` design:
the emitter retains one DOM node per persistent widget across frames (keyed by
the React fiber's stable id, mirroring Flutter's `SemanticsObject` id-to-DOM
binding), dirties fields per-update so we PATCH attributes rather than
rebuild, and composes behaviors (`Focusable`, `Tappable`, `LiveRegion`,
`LabelAndValue`) over role primitives (`ButtonRole`, `CheckableRole`,
`ScrollableRole`, `GenericRole`). See
`flutter/engine/lib/web_ui/lib/src/engine/semantics/semantics.dart` for the
canonical SemanticRole + SemanticBehavior decomposition we are copying.

`<jet-semantics>` is the **shared shadow subtree** for both the focus
channel (#2135) and this a11y channel. The split of concerns:

| Channel | Cares about |
|---------|-------------|
| focus #2135 | `tabindex`, focus order, `:focus-visible`, key event routing into WASM |
| a11y #2136 | `role`, accessible name, `aria-*` state, live-region announcements |

Both channels read from the same emitter output but assert on disjoint
attribute sets. Conflicts are resolved in the emitter contract (#2158), not
in test code.

The hidden subtree is **not** visually rendered (`position: fixed;
clip: rect(0,0,0,0); pointer-events: none`) but **is** kept in the
accessibility tree (no `aria-hidden=true`, no `display: none`). Pointer
events on the visible canvas are forwarded into WASM through the pointer
channel (#2137); pointer events that land on a hidden semantics node (e.g.
synthesized by a screen reader's virtual cursor) are translated back into
in-canvas hit-tests by the emitter so AT can still "click" buttons.

## Technical approach

- **Emitter contract (#2158).** One table, one entry per jet widget,
  three columns: ARIA role, required state attributes (`aria-checked`,
  `aria-selected`, `aria-expanded`, `aria-disabled`, `aria-valuenow`,
  `aria-valuemin`, `aria-valuemax`, `aria-valuetext`, ...), and the
  ARIA APG pattern URL the mapping conforms to
  (`https://www.w3.org/WAI/ARIA/apg/patterns/<pattern>/`). Source the
  list from MUI's per-component a11y notes — every MUI component the
  parity corpus uses has a corresponding APG pattern, so the mapping
  is mostly transcription. Composite widgets (combobox, menubutton,
  treegrid) expand into multi-element fragments per their APG examples.

- **Computed AX tree diff (#2160).** This is the strongest signal because
  CDP `Accessibility.getFullAXTree` returns the browser's *computed*
  tree — after the accname algorithm, after role mapping, after
  hidden-node ignoring — not the source DOM. Two renderers can ship
  byte-different DOM and still produce identical AX trees, which is what
  we actually want to assert. Snapshot both react-dom and jet, normalize
  each `AXNode` to `{role, name, value, properties_subset, ignored}`
  ignoring `nodeId` / `backendDOMNodeId` / geometry, and diff. A
  per-fixture whitelist (driven by foundation #2144) lets us land
  features incrementally — e.g. accept jet announcing a tooltip
  description differently from react-dom until the description
  recursion is implemented.

- **axe-core CI gate (#2159).** Run axe-core against `<jet-semantics>`
  on every fixture; gate the build on WCAG 2.1 AA violations.
  Use the same ruleset MUI's own CI uses (`mui/material-ui` runs axe via
  `jest-axe`). Fast (~seconds per fixture), high signal for
  rule-class issues axe-core catches well: missing labels, broken
  `aria-*` references, role-vs-attribute mismatches, contrast failures
  on text nodes that *do* live in the shadow DOM (image labels,
  off-screen captions). Pa11y is intentionally not added — axe-core
  is the de-facto standard and jest-axe / cypress-axe wrappers make
  it cheap to embed.

- **Live-region timing (#2161).** A toast is meaningless if announced
  10 frames late or never. The emitter must mutate `aria-live` regions
  in jet's shadow DOM at the same logical moment react-dom would (after
  commit, before next paint). Test fixtures: snackbar toast, form
  validation error appearing under a `<TextField>`, async loading state
  resolving to result. Assertion: mutation observer on
  `<jet-semantics>` records a mutation within 1 commit boundary of the
  React state transition, and the mutated subtree has `role="status"`
  or `role="alert"` per APG live-region pattern.

- **Accname computation (#2163).** Vendor a subset of
  `wpt/accname/` (foundation #2142 handles the vendoring) and run it
  against jet's emitter. The ARIA accname algorithm is fiddly —
  `aria-labelledby` references win over `aria-label`, which wins over
  native `<label for>`, which wins over visible text, which wins over
  `title` — and the recursion has corner cases (hidden subtree
  inclusion, circular ref prevention, embedded controls returning
  their value, CSS `::before`/`::after` content). WPT already encodes
  the right answers; we just need to match. Start with `name/basic.html`
  and `name/comp_label.html` style cases, expand to the full subset as
  the emitter matures.

- **Screen-reader smoke (#2162).** Automation tops out at axe + AX tree
  diff + WPT. Real AT has implementation quirks (NVDA's announcement
  buffering, VoiceOver's grouping heuristics, JAWS's virtual cursor)
  that no headless tool models. Manual matrix once per release using
  W3C ARIA-AT (`w3c/aria-at`) test plans for the patterns they cover
  (button, combobox, menubutton, etc.). Runbook captures: install AT,
  load fixture, perform documented gesture, record actual vs.
  expected announcement.

## Success criteria

- **#2158** Mapping table reviewed and signed off against APG patterns
  for every widget used in the parity corpus. Table lives at
  `projects/jet/specs/a11y-emitter-contract.md`, includes an APG
  URL column, and is the upstream contract every other sub-issue
  asserts against.
- **#2159** axe-core PASS (zero WCAG 2.1 AA violations) on all parity
  corpus fixtures (60 MUI demos at the umbrella's target corpus
  size), CI-gated, runs in under 60s for the full corpus.
- **#2160** AX tree diff against the react-dom oracle is empty for
  every fixture, modulo the per-fixture whitelist in foundation
  #2144's manifest. Whitelist entries each carry a tracking issue
  and an expiry condition.
- **#2161** Live-region announcement test: snackbar, form error, and
  async loading fixtures all emit a shadow-DOM mutation within one
  commit boundary of the React state transition, with the correct
  role (`status` / `alert`) and politeness.
- **#2162** Per-release manual smoke pass: at minimum one fixture
  exercising each ARIA-AT pattern jet implements, run on NVDA+Firefox,
  VoiceOver+Safari, JAWS+Chrome, TalkBack+Chrome-on-Android. Results
  filed as a release-gating checklist.
- **#2163** WPT `wpt/accname/` vendored subset is 100% passing.
  Subset starts at `name/basic.html` + `name/comp_label.html` +
  `name/comp_labelledby.html` and grows per release.

## Out of scope / waivers

- **Browser-native DevTools "Inspect Accessibility Properties".** It
  works on jet's shadow DOM (because that's what the browser exposes
  to AT) but it cannot point at the visible canvas pixels and tell
  you which widget is under the cursor. Acceptable divergence —
  the shadow DOM is *the* a11y truth; the canvas is just paint.
- **WCAG 1.4.4 "Resize Text" via browser zoom.** Browser text zoom
  doesn't reach into the canvas, so this success criterion is
  literally impossible to satisfy via the platform mechanism. Handled
  out-of-band via jet's own pinch-zoom / Cmd-+ binding that scales the
  canvas backing store and re-rasterizes text glyphs. Documented as a
  waiver in the conformance report, not a parity failure.
- **WCAG 1.4.12 "Text Spacing" via user CSS.** Same root cause —
  user-injected CSS doesn't reach canvas glyphs. Re-raster on a jet
  setting instead.
- **WCAG 1.4.10 "Reflow" at 320px CSS px.** jet's layout is internal,
  not driven by `@media (max-width)`. Reflow is asserted by the pixel
  channel #2134 against jet's own breakpoint set, not by browser-CSS
  reflow rules.
- **High-contrast mode / forced colors.** Windows High Contrast mode
  remaps system colors via `forced-colors` media query; jet paints
  raw pixels so the media query is meaningless. Out of scope for
  channel; tracked as a separate jet-platform-integration epic.
- **AT-specific quirk parity.** We aim to match react-dom's *AX tree*,
  not NVDA's announcement string byte-for-byte. If NVDA chooses to
  group two `role="status"` nodes that VoiceOver doesn't, that's an
  AT-vendor decision, not a jet bug. Captured in the #2162 smoke
  results as informational, not a gate.
- **`Section 508` and `EN 301 549` certifications.** Out of harness
  scope; these are legal/contractual conformance regimes that
  consume WCAG 2.1 AA + manual audits, both of which this channel
  provides inputs for but does not itself certify.

## Prior art and references

- **Flutter Web semantics**
  - `https://github.com/flutter/engine/tree/main/lib/web_ui/lib/src/engine/semantics` —
    `semantics.dart` (SemanticsObject + SemanticRole + SemanticBehavior
    architecture), `label_and_value.dart`, `live_region.dart`,
    `focusable.dart`, `tappable.dart`, `checkable.dart`,
    `incrementable.dart`, `scrollable.dart`. This is the canonical
    "canvas renderer + shadow ARIA DOM" pattern jet's emitter mirrors.
  - `https://api.flutter.dev/flutter/semantics/SemanticsBinding-class.html` —
    Flutter framework's semantics binding contract. Useful for the
    "when do we flush the dirty semantics tree" timing question.
  - `https://api.flutter.dev/flutter/rendering/RenderObject/describeSemanticsConfiguration.html` —
    per-widget contribution to the semantics tree; analogous to the
    per-jet-widget mapping rows in #2158.

- **Automated a11y testing tools**
  - `https://github.com/dequelabs/axe-core` — WCAG 2.0/2.1/2.2 A/AA/AAA
    + best practices ruleset; ~57% of WCAG issues catchable
    automatically per the project README.
  - `https://github.com/nickcolley/jest-axe` — Jest matcher
    (`toHaveNoViolations`) for axe-core. MUI's CI uses this pattern.
  - `https://github.com/component-driven/cypress-axe` — Cypress
    equivalent; useful if/when we run a11y assertions inside
    end-to-end flows rather than per-component.
  - `https://pa11y.org/` — Pa11y / Pa11y CI; CLI accessibility runner.
    Not adopted — axe-core covers the same ground and is closer to
    the de-facto standard.

- **Chrome DevTools Protocol**
  - `https://chromedevtools.github.io/devtools-protocol/tot/Accessibility/` —
    Accessibility domain.
    `Accessibility.getFullAXTree` returns the entire computed
    accessibility tree as `AXNode[]` with fields
    `{nodeId, role, name, value, description, properties, ignored,
    childIds, parentId, backendDOMNodeId}`. This is the API #2160 uses
    for the diff.

- **W3C specifications**
  - `https://www.w3.org/TR/accname-1.2/` — Accessible Name and
    Description Computation 1.2; the precedence ladder
    `aria-labelledby > aria-label > native (label / alt / desc) >
    text content > title`, plus the recursion rules #2163 must
    satisfy.
  - `https://www.w3.org/WAI/ARIA/apg/` — Authoring Practices Guide
    landing page.
  - `https://www.w3.org/WAI/ARIA/apg/patterns/` — 31 design patterns
    (button, checkbox, radio group, switch, slider, spinbutton,
    combobox, listbox, menu, menubar, menubutton, tabs, treeview,
    treegrid, accordion, disclosure, breadcrumb, landmarks,
    dialog (modal), alert, alertdialog, tooltip, grid, table,
    carousel, feed, meter, toolbar, window splitter, link).
    The #2158 emitter contract references one of these URLs per
    mapping row.
  - `https://www.w3.org/WAI/WCAG21/quickref/?versions=2.1&levels=aa` —
    WCAG 2.1 AA criteria, the rule scope for #2159.
  - `https://github.com/w3c/aria-practices` — ARIA APG source repo;
    examples directory is the practical reference for each pattern's
    DOM shape.
  - `https://github.com/w3c/aria-at` — ARIA-AT manual screen-reader
    test plans for #2162.

- **WPT (web-platform-tests)**
