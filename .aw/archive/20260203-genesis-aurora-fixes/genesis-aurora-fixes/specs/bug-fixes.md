# Genesis & Aurora Bug Fixes

## Overview

Fix 4 bugs across genesis and aurora crates: (1) Aurora Mermaid+ generators output frontmatter outside the mermaid code block, breaking platform rendering; (2) CLI impl-change has Bash permission that could cause infinite loops; (3) Specs contain outdated XState references that were never implemented; (4) plan-change workflow errors when no specs are required instead of generating simple tasks.

## Requirements

### R1 - Mermaid+ frontmatter inside code block

```yaml
id: R1
priority: high
status: draft
```

All Aurora *_plus generators must output YAML frontmatter INSIDE the ```mermaid``` code block, not outside. Current format breaks GitHub/GitLab/Jira rendering. Affected files: state_plus, flowchart_plus, sequence_plus, class_plus, erd_plus, requirement_plus, mindmap_plus, journey_plus generators.

### R2 - Remove Bash permission from impl-change

```yaml
id: R2
priority: high
status: draft
```

Remove 'Bash' from the allowed tools list in crates/cclab-genesis/src/orchestrator/claude.rs. Only MCP tools (Write, Edit, Read, Glob, Grep) should be permitted to prevent infinite loop risk where Claude subprocess re-invokes cclab commands.

### R3 - Remove XState references from specs

```yaml
id: R3
priority: medium
status: draft
```

Update specs to remove XState integration references that were never implemented. Keep Mermaid+ references which are working. Affected specs: plan-change.md, mermaid-plus-format.md, mermaid-plus-conversion.md

### R4 - Handle no-specs case in plan-change workflow

```yaml
id: R4
priority: high
status: draft
```

When a change has no specs (e.g., simple bug fixes), plan-change should generate a basic tasks.md with implementation tasks derived from the proposal, instead of erroring with 'No valid specs found'.

## Acceptance Criteria

### Scenario: Mermaid+ output renders correctly on GitHub

- **GIVEN** A Mermaid+ diagram generated by aurora_generate_state_plus
- **WHEN** The output is pasted into a GitHub markdown file
- **THEN** The frontmatter is parsed as Mermaid config, not displayed as plain text

### Scenario: impl-change cannot invoke cclab commands

- **GIVEN** Claude subprocess running impl-change workflow
- **WHEN** Claude attempts to run 'cclab' or 'cc' commands
- **THEN** The command fails because Bash tool is not available

### Scenario: Specs reflect implemented features only

- **GIVEN** A developer reads mermaid-plus-format.md
- **WHEN** Looking for XState integration details
- **THEN** No XState references exist, only Mermaid+ format is documented

### Scenario: Bug-fix changes generate tasks without specs

- **GIVEN** A proposal with no specs required
- **WHEN** Running cclab gen plan-change
- **THEN** tasks.md is generated from proposal's affected files list
