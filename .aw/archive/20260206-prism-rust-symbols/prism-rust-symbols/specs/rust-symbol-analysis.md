# Rust Symbol Analysis

## Overview

This spec defines the algorithm and data mapping for extracting semantic symbols from Rust source code within the Prism analysis engine. It extends the existing `SymbolTableBuilder` to support Rust AST nodes (generated by tree-sitter-rust), mapping them to the unified `Symbol` and `TypeInfo` structures. This enables cross-language analysis features like "Go to Definition" and "Find References" for Rust files.

## Requirements

### R1 - Extract Rust Functions

```yaml
id: R1
priority: high
status: draft
```

The system must extract function definitions (`fn_item`), including name, return type, parameters, and visibility modifiers.

### R2 - Extract Rust Structs

```yaml
id: R2
priority: high
status: draft
```

The system must extract struct definitions (`struct_item`), including name and fields (named or tuple).

### R3 - Extract Rust Traits

```yaml
id: R3
priority: medium
status: draft
```

The system must extract trait definitions (`trait_item`), including name and associated items.

### R4 - Extract Rust Impls

```yaml
id: R4
priority: medium
status: draft
```

The system must extract implementation blocks (`impl_item`), linking methods to the implemented type or trait.

### R5 - Extract Rust Constants

```yaml
id: R5
priority: low
status: draft
```

The system must extract constants (`const_item`) and static variables (`static_item`).

### R6 - Extract Rust Doc Comments

```yaml
id: R6
priority: medium
status: draft
```

The system must extract documentation comments, handling both outer (`///`) and inner (`//!`) doc styles.

### R7 - Parse Rust Types

```yaml
id: R7
priority: medium
status: draft
```

The system must parse Rust type signatures (e.g., `Option<String>`, `Vec<i32>`, `&str`) into the unified `TypeInfo` structure.

## Acceptance Criteria

### Scenario: Happy Path - Struct and Impl

- **GIVEN** A Rust file with a struct and an impl block
- **WHEN** The file is parsed and `build_rust` is called
- **THEN** The symbol table should contain symbols for `MyStruct`, `new`, and `process`, with `new` and `process` scoped under the impl/struct.

### Scenario: Function with Generics

- **GIVEN** A Rust function with generic parameters and a complex return type
- **WHEN** The function is analyzed
- **THEN** The symbol table should contain a function symbol with `TypeInfo` reflecting `fn(T) -> Option<T>`.

### Scenario: Doc Comments

- **GIVEN** A Rust file with `///` doc comments above a function
- **WHEN** The file is analyzed
- **THEN** The symbol for the function should have the doc field populated with the comment text.
