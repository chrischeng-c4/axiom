## Capability Alignment

Create a reusable Mamba-native dependency injection library for mambalibs instead of keeping DI inside `mambalibs.http`.

The first slice introduces a small `mambalibs.di` core that can be consumed by HTTP now and by agent/queue/database libraries later.

## Acceptance Criteria

- Add a dedicated DI library under `projects/mamba/mambalibs/`.
- Register a Mamba namespace for the DI surface, separate from `mambalibs.http`.
- Provide a minimal Rust core for providers, scopes, resolution, and test overrides.
- Update `mambalibs.http` documentation/specs so `Depends` is an HTTP adapter over the shared DI layer, not HTTP-owned core logic.
- Add focused tests for provider registration, scope caching, override behavior, and Mamba registry exposure.

## Implementation Evidence

- Added `projects/mamba/mambalibs/dikit` as `mambalibs-di`.
- Added `projects/mamba/mambalibs/dikit/binding` as `mambalibs-di-binding`, registering `mambalibs.di`.
- Added core `Container`, `ProviderKey`, `ScopeKind`, `RequestScope`, `Resolver`, and `DependencyMarker`.
- Added Mamba binding symbols: `Container`, `RequestScope`, `Depends`, `container_register_value`, `container_override_value`, `container_clear_override`, `container_resolve`, and `scope_resolve`.
- Updated `mambalibs.http.Depends` to record a shared `mambalibs.di::ProviderKey`.
- Added force-link coverage for `mambalibs.di` under mamba `native-modules`.
- Added TD source at `.aw/tech-design/projects/mamba/specs/3952.md`.
