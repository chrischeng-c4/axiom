# test(mamba): RLIMIT sandbox in cpython conformance runner (runner.rs)

Atom 1 of epic #3934 (retire run.py; Rust harness as sole conformance runner).

## Problem

`tests/cpython/harness/runner.rs` shells `mamba run <fixture>` with no resource
bounds. A runaway fixture (infinite loop, unbounded allocation) hangs or OOMs the
`cargo test` process itself. `run.py` already guards this via per-child POSIX
RLIMIT (`_limit_child`: RLIMIT_AS / RLIMIT_DATA / RLIMIT_CPU / RLIMIT_CORE,
best-effort on macOS). Port that guard into the Rust runner so removing run.py
loses no containment.

## Capability Alignment

Capability: cpython conformance runs are self-contained — a single malformed
fixture cannot take down the harness.
Capability Gap: the Rust `conformance` runner has no per-fixture resource cap;
only run.py does, so the runner is not yet a safe superset.
Progress Evidence: a deliberately-runaway fixture is killed by the kernel and
reported as a gating verdict, while the rest of the sweep completes.

## Acceptance Criteria

- AC1: each `mamba run` child in runner.rs is spawned under address-space, CPU,
  and core-size limits (pre-exec on unix; best-effort per axis like run.py).
- AC2: an infinite-loop fixture is terminated (CPU/timeout) and surfaces a
  TIMEOUT/CPU verdict instead of hanging the test binary.
- AC3: an unbounded-allocation fixture is contained (MemoryError/OOM) instead of
  OOM-killing the host.
- AC4: the existing passing sweep (`cargo test -p mamba --test conformance`) is
  unchanged (no new failures from the caps).
