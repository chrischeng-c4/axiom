# ADR-030: MUI reference corpus snapshot — 5-channel artifact bundle per fixture × (browser, os, dpr), byte-equivalent re-run gate

| Field | Value |
|-------|-------|
| Issue | #2151 |
| Parent epic | #2134 |
| Status | accepted |
| Date | 2026-05-16 |
| Decision | The MUI reference corpus is captured by running the DOM reference runner (ADR foundation #2139, jet-parity-oracle) once per `(fixture, browser, os, dpr)` combination and emitting a fixed 5-artifact bundle: `screenshot.png` (pixel baseline), `ax-tree.json` (normalized CDP `Accessibility.getFullAXTree` per ADR-008/#2160), `focus-trace.json` (Tab/Shift+Tab sequence per ADR-003/#2153), `pointer-hit-map.json` (1000-click hit-test per ADR-009/#2165), and `ime-trace.json` (composition event protocol per ADR-013/#2172 — placeholder rows until #2173 lands). ## Context

The parity channel (#2134) needs a *concrete*, falsifiable
reference. Saying "jet should match MUI" is unfalsifiable until
we pin a specific MUI version at a specific React version, render
a specific JSX through the DOM reference runner from #2139, and
freeze the resulting set of artifacts as the canonical *expected*
set. Every downstream comparator — the pixel ladder in #2147,
the AX-tree diff in #2160, the focus-trace diff in #2153, the
hit-test diff in #2165, the IME trace diff in #2172 — needs
those frozen artifacts to compare against. Without them the
parity gate (#2144) has nothing to gate on.

This ADR is the *snapshot capture protocol*. It is the contract
that turns "the oracle output" (a runtime byproduct of running the
oracle once) into "the committed baseline" (a checked-in, reviewed,
versioned artifact tree that the CI gate compares against on every
PR). It pins five things:

1. **What** the oracle emits per `(fixture, browser, os, dpr)` —
   the 5-artifact bundle.
2. **Where** those artifacts live in the repo — the
   `goldens/<channel>/<browser>-<os>-<dpr>/<fixture>/` layout.
3. **How** snapshots are generated and regenerated — the
   `score parity oracle` CLI surface and its `--update-baseline`
   companion.
4. **The determinism gate** — re-running the oracle on a clean
   checkout MUST produce byte-equivalent output. CI enforces this.
5. **The cache-invalidation rule** — pinning MUI + React in the
   fixture's `package.json`, and the corpus-wide re-baseline that
   a version bump triggers.

The companion ADRs already define the *individual channel*
contracts: ADR-002 (per-renderer baselines, the directory
roots), ADR-008/#2160 (AX-tree normalization), ADR-003/#2153
(focus-trace protocol), ADR-009/#2165 (hit-test fixture),
ADR-013/#2172 (IME composition protocol), ADR-015 (font-loading
determinism), ADR-001/#2145 (Playwright launch flags), ADR-012/#2149
(DPR + LFS budget). This ADR composes them: one fixture, one
oracle invocation, five files out, indexed by the cross-product
of all parity axes, gated by byte-equivalent re-run.

#2150 owns the *corpus manifest* (which fixtures, which MUI
demos, which JSX). This ADR owns the *snapshot of that manifest*
(what files those fixtures produce when the oracle runs them).
The two are separate so the manifest can grow (add a fixture,
add a tier-b region tag) without re-defining the artifact
contract, and so this artifact contract can evolve (add a
sixth artifact, change a JSON shape) without re-curating the
fixture list.

#2139 defined the *runner* (the oracle process: a stock React +
ReactDOM + MUI render harness, driven by Playwright, that emits
the 5 channels' raw output). This ADR defines the *snapshotting*
of that runner's output: how the raw bytes become the canonical
expected set, where they get stored, and how the
re-run-equivalence gate is enforced. Picking the wrong line
between #2139 and this issue would either bake the storage
layout into the runner (making the runner non-reusable for
non-snapshot workflows like ad-hoc diff exploration) or leave
the snapshot layout undefined (making CI's pass/fail check
ambiguous). The split here puts *production* in #2139 and
*persistence* here.

## Decision

### Per-fixture artifact bundle (5 channels)

Each invocation of `score parity oracle <fixture>` for one
`(browser, os, dpr)` combination produces exactly 5 files:

| Artifact | Format | Owner ADR | Purpose |
|----------|--------|-----------|---------|
| `screenshot.png` | PNG (LFS) | ADR-015, ADR-024 | Pixel baseline; consumed by #2147 tier-a/b/c/d comparator and #2150 diff viewer. |
| `ax-tree.json` | canonical JSON | ADR-008 / #2160 | Normalized CDP `Accessibility.getFullAXTree`; consumed by AX-tree diff. |
| `focus-trace.json` | canonical JSON | ADR-003 / #2153 | Tab/Shift+Tab traversal sequence + roving focus events. |
| `pointer-hit-map.json` | canonical JSON | ADR-009 / #2165 | 1000-click grid hit-test, sub-pixel-quantized. |
| `ime-trace.json` | canonical JSON | ADR-013 / #2172 | Composition event sequence; placeholder rows until #2173 manual matrix lands. |

The bundle is *atomic*: either all 5 files are written for a
given `(fixture, browser, os, dpr)` cell, or none are. The CLI
writes to a temp dir and renames once all 5 are present and
have passed canonicalization. Partial bundles are a CI error.

### Storage layout

```
projects/jet/data/parity/goldens/
  <channel>/
    <browser>-<os>-<dpr>/
      <fixture>/
        screenshot.png        # LFS-tracked
        ax-tree.json
        focus-trace.json
        pointer-hit-map.json
        ime-trace.json
```

Channel roots (`pixel/`, `ax/`, `focus/`, `hit-test/`, `ime/`)
follow ADR-002's per-channel directory convention so a comparator
can scope itself to its channel without traversing siblings. The
`<browser>-<os>-<dpr>` segment uses the canonical form
`{chromium,firefox,webkit}-{linux,macos,windows}-{1.0,1.5,2.0}`
per ADR-012 (#2149); the 18 combinations are pinned and any
deviation (e.g. `chrome` vs `chromium`) is a lint error.

Fixture slug (`<fixture>`) is the same string used as the row
key in #2150's manifest — kebab-case, version-suffixed
(`mui-button-primary-v1`). Bumping the suffix (`-v1` → `-v2`)
is the explicit signal that a fixture's JSX has changed and the
old goldens are obsolete; the v1 directory is deleted in the
same PR.

### Canonical JSON

All 4 JSON artifacts MUST be serialized with:

- Keys sorted lexicographically at every depth.
- 2-space indentation, trailing newline.
- Floats rounded to 4 decimal places (ADR-008 rule for AX
  bounding boxes; applied uniformly across all JSON files for
  consistency).
- No embedded timestamps. No process IDs. No host machine
  identifiers. No path-absolute strings. Run-specific metadata
  (Playwright session ID, oracle git SHA at capture time) lives
  in a separate sidecar `manifest.json` at the
  `<browser>-<os>-<dpr>/<fixture>/` level, which is *not* part
  of the determinism gate and *not* required to be byte-equal
  across runs. The 5 artifacts above are.

The canonicalization step is a library function
(`score::parity::canonical_json`) shared by every artifact
emitter so the rule is enforced at one point, not 5.

### CLI surface

```bash
# capture (or re-verify) one fixture across all 18 (browser, os, dpr) cells
score parity oracle <fixture>

# capture every fixture in the corpus
score parity oracle --all

# regenerate the baseline for one fixture (writes over existing files)
score parity oracle --update-baseline <fixture>

# regenerate the entire corpus (used after MUI/React version bump)
score parity oracle --update-baseline --all

# determinism gate (CI): re-run oracle on a clean checkout, assert
# byte-equivalent output to the committed baseline; exits non-zero on drift
score parity oracle --verify
score parity oracle --verify <fixture>
```

`--verify` is the gate the parity CI job (#2144) runs. It is
*not* the same as the per-channel comparator runs: it asserts
the *oracle itself* is deterministic against the committed
baseline, before any comparator gets a chance to compare jet's
output against it. Drift detected by `--verify` means the
*oracle* is non-deterministic (font subset moved, MUI source
moved, React render-pass order moved), not that jet is broken.
That distinction is the reason this is a separate gate from
the channel comparators.

### Determinism gate (the central invariant)

> Re-running `score parity oracle <fixture>` on a clean
> checkout of the same git SHA MUST produce byte-identical
> output to the committed baseline. Always. For every fixture,
> every channel, every `(browser, os, dpr)` cell.

This is the gate. It is enforced by:

- **Font determinism** (ADR-015 / #2148): the oracle's
  font-loading preamble is byte-identical to jet's; the same
  woff2 subsets are loaded in the same order with the same
  unicode ranges. A font cache miss does not change emitted
  bytes.
- **Playwright launch flags** (ADR-001 / #2145): the launch
  arg set is frozen per `(browser, os)` pair; flags live in
  `projects/jet/data/parity/fixtures/playwright-flags.json` and
  changing them is a corpus-wide re-baseline event.
- **Canonical JSON** (above): no source of nondeterminism in
  the JSON itself.
- **CDP capture ordering** (ADR-008): the AX-tree is fetched
  with `interestingOnly: false` and walked depth-first
  left-to-right; the order is deterministic from the CDP
  contract.
- **Hit-test grid** (ADR-009): the 1000 click points are
  generated from a fixed PRNG seed derived from the fixture
  slug, so the same fixture produces the same grid forever.
- **Composition event protocol** (ADR-013): the IME trace is
  driven by a scripted CDP `Input.dispatchKeyEvent` sequence,
  not a real IME; sequence is deterministic by construction.

A `--verify` failure is therefore a **bug**, not a flake. We
do not retry. We do not relax thresholds. We find which of the
six determinism inputs above changed and pin it back.

### Update flow

```
# author bumps MUI version OR changes a fixture's JSX
$ vim projects/jet/data/parity/fixtures/mui/package.json
$ pnpm install
$ score parity oracle --update-baseline --all

# review the diff
$ git status
  goldens/pixel/chromium-linux-1.0/mui-button-primary-v1/screenshot.png  (LFS)
  goldens/ax/chromium-linux-1.0/mui-button-primary-v1/ax-tree.json
  ... (270 files for the initial 3-fixture corpus)

# PR
$ git add projects/jet/data/parity/fixtures/mui/package.json
$ git add projects/jet/data/parity/goldens/
$ git commit -m "chore(jet): bump MUI 6.1.3 → 6.1.4 + re-baseline corpus"
```

PR reviewer sees:

- The PNG diffs via #2150's pixel diff viewer (rendered inline
  in the PR check). Each diff is one click.
- The JSON diffs via the standard GitHub three-way text diff,
  rendered as side-by-side because the JSON is canonical and
  line-stable.
- The `package.json` diff showing the intent (which versions
  moved).

An MUI patch bump that produces zero artifact changes is still
required to be a baseline-update PR — we never accept "version
moved but I trust nothing changed". The `--verify` gate catches
the case where the author forgot to re-baseline.

### Cache invalidation

The fixture directory `projects/jet/data/parity/fixtures/mui/` owns
its own `package.json` pinning:

```json
{
  "dependencies": {
    "@mui/material": "6.1.3",
    "react": "18.3.1",
    "react-dom": "18.3.1"
  }
}
```

Bumping `@mui/material`, `react`, OR `react-dom` triggers a
corpus-wide re-baseline. No exceptions. The re-baseline is a
single atomic PR (one commit, all 270 files moved together) so
git history preserves the linkage "MUI bump → these artifacts
changed".

No other dependency in the fixture's `package.json` is allowed
to influence oracle output. The `--verify` gate catches any
violation (a transitive dependency that moves the bytes is a
bug we fix by pinning).

### LFS strategy

- **PNGs to LFS**: per ADR-012 / #2149's 32MB total budget.
  Each `screenshot.png` averages ~150 KB; 270 PNGs = ~40 MB
  uncompressed, but LFS deduplication on re-baselines keeps
  the actual storage growth bounded.
- **JSON in-tree**: all 4 JSON channels stay in-tree as
  regular blobs. They are small (<50 KB each, typically much
  less), diffable, and three-way-mergeable in case two PRs
  re-baseline overlapping fixtures. The total in-tree JSON
  footprint for the initial 3-fixture corpus is ~10 MB across
  216 files; for the full 30-fixture corpus (#2150 final
  state) it scales to ~100 MB. Still under the in-tree budget;
  no need to move JSON to LFS.

### Initial corpus

At first green, three fixtures land:

- `mui-button-primary-v1`
- `mui-textfield-outlined-v1`
- `mui-checkbox-basic-v1`

× 5 channels × 18 `(browser, os, dpr)` cells = **270 artifacts**.

Subsequent fixtures (the 9 additional in #2150's first
expansion: `Dialog`, `Menu`, `Switch`, `Slider`, `Tooltip`,
`Snackbar`, `Tabs`, `Card`, `AppBar`) land in follow-up PRs,
each adding 90 artifacts (5 × 18). The full 30-fixture target
from epic #2134 R8 is 2700 artifacts; well inside ADR-012's
LFS budget when amortized across the channel.

## Consequences

### Positive

- **Falsifiable.** "Jet matches MUI" becomes a byte-for-byte
  claim against a committed file tree. Pass/fail is mechanical;
  no human in the loop on the gate.
- **Reviewable.** PR diff surfaces every regression — pixel
  drift via inline viewer, JSON drift via text diff. A
  reviewer can approve "yes this version bump moved the
  TextField border 1 px and that's expected" without needing
  to re-run anything locally.
- **Re-runnable.** `score parity oracle --update-baseline` is
  one command. No magic environment, no manual file shuffling.
  The same command runs in CI on the re-baseline PR's check
  to verify the author actually re-ran it (the `--verify`
  gate).
- **Deterministic by construction.** Six independent sources
  of nondeterminism (font load, launch flags, JSON serial,
  CDP order, hit-grid seed, IME script) are each pinned by a
  named contract. Drift in any one is a localized fix.
- **Decoupled from comparators.** The pixel comparator (#2147),
  AX-tree diff (#2160), focus diff (#2153), hit-test diff
  (#2165), and IME diff (#2172) can each evolve independently
  — they all consume the same `goldens/<channel>/...` tree
  this ADR defines.
- **Atomic version bumps.** A MUI version move is one PR with
  one set of artifact changes. No "we forgot to re-baseline
  cell X" footgun; `--verify` catches it.

### Negative

- **LFS storage growth.** 30 fixtures × 18 cells × 1 PNG =
  540 PNGs per re-baseline. LFS dedupes unchanged blobs but
  a full MUI major bump can churn most of them. ADR-012's
  32 MB budget is a soft ceiling; if we breach it the
  retention policy is to evict per-fixture history beyond
  the last 3 baselines.
- **Re-baseline PRs are large.** A MUI patch bump is at least
  a 270-file PR (initial corpus) and grows to 2700-file PRs
  at full corpus. Reviewers need #2150's diff viewer
  integration to make this tractable; reviewing 2700 raw
  diffs is not a thing.
- **Determinism is fragile.** Every new artifact, every new
  channel, every new normalization rule has to be added with
  the canonicalization contract in mind. A regression in
  `canonical_json` (e.g. accidentally emitting a timestamp)
  silently invalidates the entire gate. Mitigated by the
  `--verify` job running on every PR.
- **18-cell matrix is expensive.** Capturing 18 cells per
  fixture means 18 Playwright launches per fixture per
  re-baseline. At ~20 s per launch (oracle + 5 captures +
  serialize), one fixture takes ~6 min on a single-core run;
  the full corpus re-baseline takes ~3 hours linear. Mitigated
  by `--update-baseline --all` running cells in parallel
  (#2145's Playwright worker pool).
- **Placeholder IME trace.** Until #2173's manual IME matrix
  lands, `ime-trace.json` carries only scripted-event rows
  and not real CJK composition. Reviewers must know not to
  read too much into IME diffs in the interim.

## Alternatives considered

### Alt A: store only PNGs, derive JSON channels at compare time

Run the oracle live during the CI compare step, capture all 4
JSON channels fresh, diff against jet's output. No `goldens/*.json`
committed; only the PNGs are baselined.

Rejected: live oracle on every PR is ~3 hours per run at full
corpus (see Negative §4). The whole point of a *snapshot* is to
move that cost out of the hot path. Also loses three-way merge
on JSON conflicts.

### Alt B: monolithic snapshot file

Concatenate all 5 channels into a single `bundle.tar.zst` per
fixture per cell.

Rejected: kills text-diffability. The PR review loop depends on
GitHub rendering JSON diffs inline; tar files are opaque blobs.
Also loses partial-update semantics — bumping just the AX
emitter shouldn't require touching the PNG.

### Alt C: separate repo for goldens

Push `goldens/` to a sibling repo (`cclab/jet-parity-goldens`)
to keep the main repo lean.

Rejected: breaks atomic PRs. A change that affects both source
(oracle code) and golden (oracle output) would need a
two-repo dance; the cross-repo coordination is the bug we'd
ship over and over.

### Alt D: no determinism gate, just compare-and-warn

Accept that the oracle is approximately deterministic, gate
only on the channel comparators (#2144 parity gate), and let
small drift through as "noise".

Rejected: defeats the corpus's purpose. If the oracle output
shifts by 1 px between runs we no longer know whether jet
*matches* MUI or just matches "MUI as captured during this
particular run". The whole channel collapses into flake-watching.

### Alt E: skip `--verify` in CI; trust the author

Make `score parity oracle --update-baseline` the only enforcement
point — author runs it locally, commits the output, reviewer
trusts it.

Rejected: the failure mode is silent (someone forgets to
re-baseline a cell after a font subset change) and shows up as
a parity-gate regression weeks later on an unrelated PR. The
`--verify` job is cheap (~3 min on the standard runner using a
warm font cache) and catches the bug at the source.

## Open questions

- **Cross-fixture font cache.** Currently every `(browser, os, dpr)`
  cell does a cold font load. Can we share a warmed font atlas
  across cells in the same `--all` run without breaking
  determinism? Likely yes (the cache hits are byte-equivalent
  reads) but needs an ADR-015 sub-clause to lock the contract.
- **Drift detection on transitive deps.** A move in MUI's
  `@emotion/react` peer dep can shift bytes even when
  `@mui/material` itself is pinned. Today's gate catches the
  shift but not the *cause*. A lockfile diff sidecar in the
  re-baseline PR would help reviewers triage.
- **IME real-trace integration.** #2173's manual CJK matrix
  produces real composition traces. The `ime-trace.json`
  contract here is the placeholder; when #2173 lands we'll
  need a one-time migration PR re-baselining every fixture's
  IME channel.
- **Headless vs headed parity.** The oracle runs headless;
  some MUI text rendering (specifically subpixel AA at DPR=1.0
  on Linux) differs between headless and headed Chromium. Not
  in scope for this ADR (the goldens are headless by
  definition) but worth tracking against the channel comparator
  in #2147 so jet doesn't get blamed for headless-vs-headed
  delta.
- **Corpus-wide re-baseline cadence.** MUI cuts a patch every
  ~2 weeks. Do we re-baseline on every patch (high churn) or
  pin to minors (less churn, more drift between MUI features
  and our coverage)? Currently undefined; this ADR allows
  either policy.
